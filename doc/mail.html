mail

 
using a HTTP interface to messages stored as files referenced by their message IDs is the overview.

to put these files on a filesystem, you could use fetchmail, getmail or offlineimap.

to avoid scanning an entire corpus of messages to look for updates, date-dirs are filled. this procmailrc contains the strftime template

a shell script automates fetch/indexing, to be put in /etc/cron.hourly or run on demand

inbox URLs:



http://m/t?view=page&v=threads&c=8 newest messages


http://m/.mail?view=threads&y=day all of today's mail


http://m/search?q=sioc+ non-email messages




if TMail is throwing errors , gem install -v 2.3.5 actionmailer, and require or move its vendor/tmail* to ruby's path, eg output of ruby -rrbconfig -e 'puts Config::CONFIG["sitelibdir"]'

suggestions welcome for other FFI/C-based mail parser libraries. pure-ruby options were 2 orders magnitude slower, fine for generating 'thank you for joining site ____' in background process, when parsing hundreds of mail to generate inbox was unusable. i'll proably actually finish entire port of project to Haskell first, started this before discovering TMail..
